# MVP_SCOPE.md

## 文档状态

- 项目：小奶瓶 / 宝宝成长记录
- 阶段：MVP 范围初版
- 日期：2026-05-28

## 已确认事实

1. 第一版只做 iOS 原生 App。
2. 目标用户以妈妈为主，宝宝年龄 0-3 岁。
3. 第一版免费验证，不商业化。
4. 单人记录进入第一版，多人协作不进入第一版。
5. 照片可以复制进 App 私有空间。
6. 第一版需要账号。
7. 第一版需要备份恢复。
8. 服务器需要存储宝宝照片原图。
9. 首发地区为香港和美国。
10. 疫苗提醒模板覆盖国内 + 香港。
11. 崩溃上报进入第一版，使用 Apple 原生渠道。
12. API 服务与对象存储区域需在发布前按合规策略确认。

## 合理推断

1. MVP 应围绕每日高频记录和回看价值，而不是完整育儿平台。
2. 第一版要压低输入成本，细节字段应默认可跳过。
3. 后端存储进入 MVP 后，应只做第一方账号、备份恢复和照片原图文件存储，不扩展成管理后台或 BI。

## 待我确认的问题

1. 云端数据保留周期和删除 SLA。
2. 小奶瓶 API 正式域名。
3. 付费 Apple Developer 账号开通时间。
4. 是否需要 iPad 兼容作为第一版验收项。

## 第一版必须做

| 功能 | 为什么必须做 | 用户频率 | 验收标准摘要 |
|---|---|---:|---|
| 宝宝档案 | 所有记录需要归属到宝宝 | 低 | 可创建、编辑、删除；可记录昵称/生日/性别可选/头像可选 |
| 今日首页 | 高频入口和今日状态需要集中 | 高 | 能看到今日记录摘要、宝宝出生天数、快速记录入口 |
| 纪念日 | 增强持续记录和纪念感 | 中 | 自动展示出生第几天、满月、百天等关键节点 |
| 喂养记录 | 新生儿高频刚需 | 高 | 可记录时间、类型、数量/时长可选、备注可选 |
| 睡眠记录 | 日常节律核心数据 | 高 | 可记录开始/结束，支持未结束睡眠状态 |
| 排便记录 | 新生儿健康观察常用记录 | 高 | 可记录时间，颜色/形态/备注作为可选项 |
| 身高体重记录 | 成长趋势核心数据 | 低 | 可记录身高、体重、日期，能展示历史列表和趋势入口 |
| 宝宝照片时间线 | 形成纪念册感和回看价值 | 中 | 可从相册选择或拍照导入，复制到 App 私有空间，按时间展示 |
| 疫苗提醒 | 低频但重要 | 低 | 可新增提醒、编辑、删除、完成标记；模板覆盖国内 + 香港；不提供医疗建议 |
| 月度成长报告 | 给持续记录一个回报 | 月度 | 能按月份聚合记录、照片和成长数据生成报告 |
| 本地删除机制 | 敏感数据必须可控 | 中 | 用户可删除单条记录、照片、宝宝档案及相关数据 |
| 账号 | 备份恢复需要稳定身份 | 中 | 可登录、可退出、可删除账号；账号方式待确认 |
| 备份恢复 | 用户已确认需要云端备份恢复 | 中 | 可备份本地记录、可恢复到同一账号；断网不影响本地使用 |
| 照片原图云备份 | 用户已确认服务器存照片原图 | 中 | 只备份用户主动加入 App 的照片；可删除云端原图；隐私说明明确 |
| 服务器存储规则 | 用户已确认后端存储诉求 | 中 | 在实现前明确用途、字段、加密、删除、隐私标签 |
| 崩溃上报 | 真实上线需要稳定性证据 | 中 | 不采集宝宝内容、照片、对象 key、记录明细；SDK 清单与隐私标签一致 |

## 第一版不做

| 功能 | 为什么先不做 | 后续触发条件 |
|---|---|---|
| macOS 端 | 用户已确认第一版只做 iOS | iOS 版验证留存后 |
| Android / Web | 分散原生体验和测试成本 | 明确跨平台商业需求 |
| 家庭多人协作 | 会引入账号、权限、冲突和隐私复杂度 | 单人记录验证成功，用户强需求明确 |
| 亲友共享/公开分享 | 增加儿童照片传播风险 | 完成隐私设计和分享权限模型 |
| 社区 | 偏离记录工具核心 | 有明确内容运营策略 |
| 医疗诊断/AI 诊断 | 审核、责任和安全风险高 | 有专业合规支持 |
| 电商/广告/订阅 | 第一版免费验证 | 留存验证成功后另开商业化规格 |
| 第三方分析 SDK | 儿童和家庭数据敏感 | 隐私审查通过且有不可替代理由 |
| 完整后台管理系统 | 第一版不需要重后端 | 服务器存储规模化后 |
| 自动识别照片/健康数据 | 权限和隐私风险高 | 用户价值明确且隐私审查通过 |
| 中国大陆首发 | 首发已确认香港和美国 | 备案、ICP、主体和儿童数据合规准备完成后 |

## 砍功能原则

1. 不能帮助妈妈完成每日记录和回看的，先砍。
2. 会显著增加账号、协作、同步、合规复杂度的，先砍。
3. 需要第三方上传儿童或家庭数据的，先砍。
4. 高频记录优先，低频管理后置。
5. 情绪价值服务于记录，不做纯装饰功能。
