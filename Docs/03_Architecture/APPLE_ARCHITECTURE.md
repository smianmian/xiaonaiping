# APPLE_ARCHITECTURE.md

## 文档状态

- 项目：小奶瓶 / 宝宝成长记录
- 阶段：Apple 架构更新
- 日期：2026-05-29
- 实现状态：已创建 `App/iOS/XiaoNaiPing.xcodeproj`；当前只有 SwiftUI 工程骨架、本地 mock 数据和占位 UI，未接入后端、账号、照片权限、CloudKit 或第三方依赖

## 已确认事实

1. 第一版只做 iOS 原生 App。
2. 技术方向是 Swift + SwiftUI。
3. 本地优先，优先评估 SwiftData / Core Data。
4. 第一版需要账号。
5. 第一版需要备份恢复。
6. 服务器需要存储宝宝照片原图。
7. 首发地区为香港和美国。
8. 疫苗提醒模板覆盖国内 + 香港。
9. 崩溃上报进入第一版。
10. 崩溃上报使用 Apple 原生渠道。
11. API 服务器和对象存储第一版默认按合规策略确认。
12. macOS 第一版不做。
13. 当前 iOS 工程部署目标设置为 iOS 18.0。

## 合理推断

1. 第一版架构应按 iOS-only 设计，但保留未来扩展空间。
2. 本地持久化应先稳定，再考虑上传。
3. 后端服务器存储需要独立 API、账号、对象存储、隐私审查和删除机制后才能实现。
4. CloudKit 暂不作为默认方案，但保留为未来替代方案。

## 模块边界

1. App Shell：启动、导航、Tab、全局状态。
2. Baby Profile：宝宝档案和纪念日计算。
3. Daily Logging：喂养、睡眠、排便。
4. Growth：身高体重、疫苗提醒、月报。
5. Photo Archive：照片导入、私有空间存储、时间线。
6. Persistence：本地数据库、照片文件管理、迁移。
7. Account：登录、退出、删除账号、凭证存储。
8. Sync：服务器备份/恢复、照片上传/下载、删除重试。
9. Privacy：权限说明、数据删除、隐私设置。
10. Diagnostics：崩溃上报、脱敏、稳定性证据。

## 数据流

1. 用户操作先写入本地模型。
2. UI 从本地数据生成今日摘要、时间线和月报。
3. 本地变更进入备份队列，网络可用时上传服务器。
4. 同步失败不得影响本地记录。
5. 删除操作必须同时覆盖本地数据、本地照片文件、服务器记录和云端照片对象。

## 依赖策略

1. 第一版优先使用 Apple 原生框架。
2. 不引入第三方 UI 框架。
3. 不引入第三方分析、广告、用户画像 SDK。
4. 任何依赖必须先写 `DEPENDENCY_POLICY.md`。

## 风险点

1. 后端服务器存储会引入账号、安全、隐私政策和删除机制。
2. 照片原图上传服务器风险高，必须做私有对象存储、删除验证和隐私披露。
3. 疫苗提醒不能写成医疗建议。
4. SwiftData/Core Data 的选择会影响迁移和测试策略。
5. 崩溃日志不得包含宝宝隐私内容、照片对象 key 或服务器 token。

## 替代方案

1. 纯本地：隐私最强，上线更快，但换机恢复弱。
2. CloudKit：Apple 生态一致，但当前已确认第一方服务器存储。
3. 第一方服务器备份：当前选定方向，满足账号恢复和原图备份，但需要后端和隐私能力。
